# Podcast Automator Constitution

## コア原則

### I. TS/Pythonのクリーンコード

読みやすさと保守性を最優先する。TypeScript/Pythonともに小さく責務が
明確な関数・クラスに分割し、重複や過度な抽象化を避ける。型情報・型注釈、
命名、フォーマッタ/リンタの規約を遵守し、将来の変更コストを最小化する。

### II. ユニットテスト必須（非交渉）

TypeScript/Python双方で、変更・追加したロジックにはユニットテストを必ず
追加する。テストが失敗する状態から実装を行い、最終的に全テストが通る
ことを完了条件とする。

### III. 一貫したUI/UX

フロントエンドのUI/UXは既存のデザインパターンに合わせ、色、余白、タイポ
グラフィ、コンポーネントの振る舞いを統一する。新規UIは既存の一貫性を
損なわない範囲で追加し、破壊的な差異は設計合意の上で行う。

### IV. 日本語のドキュメントとIssue運用

仕様、設計、Issue、PR説明、運用メモは日本語で記述する。決定事項・前提・
理由を明確に残し、チーム内外の理解コストを下げる。

## 技術スタックと品質基準

- フロントエンド: TypeScript。
- バックエンド: Python。
- コード品質: プロジェクト標準のフォーマッタ/リンタに準拠する。
- テスト: 既存のテスト基盤に合わせ、ユニットテストを必須とする。
- UI/UX: 既存画面・コンポーネントのパターンを基準にレビューする。

## 開発ワークフロー

- 変更はIssueを起点とし、背景・目的・受け入れ条件を日本語で明記する。
- 仕様/設計/タスクはテンプレートに従い、憲法チェックを通過する。
- PRには影響範囲、テスト結果、UI変更の有無を日本語で記載する。
- レビューではコア原則（クリーンコード、ユニットテスト、UI/UX一貫性、
 日本語ドキュメント）への適合を確認する。

## ガバナンス

- 憲法は他の開発慣行やテンプレートに優先する。
- 改定は「提案 → 影響範囲 → 具体的な変更 → 移行手順」を文書化し、
 レビュー合意後に反映する。
- 版管理はセマンティックバージョニングに従う。
 	- MAJOR: 原則の削除/大幅な再定義など後方互換性の破壊
 	- MINOR: 原則/セクションの追加、実質的な拡張
 	- PATCH: 文言の明確化、誤字修正など意味変更なし
- 全てのPRで憲法チェックを実施し、例外は理由と代替案を明記して承認を得る。
